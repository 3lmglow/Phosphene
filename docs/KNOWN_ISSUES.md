# 已知问题

本文记录 2026-07-19 全仓审阅中已经复现或由代码路径确认、但不适合当作“小修”直接改动的事项。它们不会阻止普通任务、积分、奖励、网站或 MCP 使用，但会影响特定安全边界和运维场景。

## 1. 带图片的 ZIP 无法可靠地在原实例就地恢复

网站能够导出包含任务图片的 ZIP，但把同一份 ZIP 恢复到仍保留原图片文件的实例时，会因为对象 Key 已存在而失败。没有图片的备份已有自动化往返测试；带图片恢复不应视为已验证能力。

在修复完成前：

- 升级和灾难恢复以 `/data` 持久卷快照为主；
- 不要删除唯一的旧卷后再尝试未经演练的 ZIP 恢复；
- 若要迁移，先在隔离的临时实例演练，并保留源实例和卷快照。

## 2. 大型 ZIP 导出与恢复可能超过小内存实例上限

当前 ZIP 生成、上传与解析主要在内存中完成，而生产容器主动限制了 Node.js 堆。图片较多时，导出或恢复可能造成进程退出并被平台重启。

在修复完成前：

- 使用 Zeabur 卷快照保存大体积实例；
- 不要在业务高峰执行大型 ZIP 操作；
- 不要把 512 MB 上传限制理解为 512 MB 文件一定能在当前内存配置中安全恢复。

## 3. ZIP 没有恢复幂等记录

业务写操作依靠幂等 Key 防止网络重试造成重复发分、扣分、任务或兑换。当前 ZIP 不包含
`idempotency_keys`：恢复到较早时间点后，现实例中较新的 Key 仍然存在；迁移到新实例时，
旧 Key 又不会随业务数据迁移。两种情况都可能让恢复后的重试结果与业务表不一致。

在修复完成前，ZIP 恢复后不要让客户端自动重放恢复前的待处理写请求；重新连接 AI 客户端，
并为真正的新操作生成新的幂等 Key。

## 4. ZIP 导出不是同一数据库快照

导出会依次读取多张表，再读取图片对象；当前没有把整个读取过程固定在一个一致性快照中。
如果导出期间刚好发生任务提交、审核、兑换或图片写入，归档中的表和文件可能来自略有差异的
时间点。

在修复完成前，导出 ZIP 时暂停 AI 写操作，也不要同时提交任务或兑换；重要备份仍使用平台
提供的整卷快照。

## 5. AI 判定任务失败的扣分没有完整复用手工扣分边界

`adjust_points(kind="penalty")` 会检查 user 的暂停开关和每日扣分上限；`manage_task(action="fail")` 的任务扣分走另一条结算路径，目前没有完整应用这两项限制。系统自动逾期扣分本来就应与 AI 手工扣分区分，但 AI 主动判失败也需要受 user 边界控制。

在修复完成前，AI 提示词应明确：暂停惩罚时不得调用 `fail` 施加扣分，并避免在一天内连续判定多个任务失败。

## 6. 页面时间按设备时区显示

服务端按 Phosphene 设置中的时区结算自然日、连击和截止规则，但浏览器目前使用设备时区格式化时间。两者不一致时，积分结算仍以服务端为准，页面显示的时刻可能有偏差。

在修复完成前，让手机或电脑系统时区与 Phosphene 设置保持一致，尤其是在创建或处理有截止时间的任务时。

## 安全提示

备份恢复只应接收自己导出的可信 ZIP。本次审阅还确认恢复路径需要进一步做对象 Key 白名单、归档大小和一致性加固；在完成安全修复前，不要上传任何第三方提供或经过修改的备份文件。
