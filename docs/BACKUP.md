# 备份与恢复

Phosphene 同时提供整卷快照和网站 ZIP。两者解决的问题不同，正式使用时建议同时保留。

## 两种备份分别保存什么

| 方式 | 保存内容 | 不保存或不单独处理的内容 | 适合用途 |
| --- | --- | --- | --- |
| `/data` 持久卷快照 | SQLite、WAL/SHM、图片、临时恢复目录 | 由部署平台管理 | 升级前保护、完整灾难恢复 |
| 网站 ZIP | 业务设置、任务、提交、积分、奖励、成就、审计、幂等记录和图片 | 登录密码、网站会话、AI Token | 日常下载、迁移业务数据、回到某个业务时间点 |

ZIP 不覆盖登录凭证是有意设计：恢复完成后仍使用当前实例的密码、会话和 AI Token。若要把
整个实例连同凭证一起原样回滚，应使用 `/data` 卷快照。

## 从网站导出

1. 登录网站，打开“设置 → 数据与备份”。
2. 点击导出并把 ZIP 下载到本机或另一个独立存储位置。
3. 确认浏览器下载完整，再给文件标注日期和对应实例。
4. 不要只把备份留在同一 Zeabur 卷中；服务、卷和备份位于同一故障域时不能互相兜底。

导出采用数据库一致性快照，并直接把 manifest 与图片写入下载流，不会把完整 ZIP 一次性放进
Node.js 内存。导出开始前会等待正在处理的业务请求结束；导出期间 `/healthz` 仍可用，其他
业务请求会短暂返回 `503 backup_maintenance`。

当前 ZIP 格式为 version 2，包含幂等记录。Phosphene 仍可导入旧版 version 1 ZIP；旧格式
本身没有幂等记录，因此恢复旧备份时，目标实例现有的幂等记录会被清空，以免留下与已回滚
业务数据不一致的响应。

## 恢复前检查

恢复是高影响操作。开始前逐项确认：

- ZIP 来自你自己的 Phosphene，且没有被第三方修改；
- 当前实例已经另做一份 `/data` 卷快照；
- 记得当前网站登录密码；
- 没有人正在提交任务，AI 也没有待发送写操作；
- `/data` 有足够临时空间。

恢复时，上传的 ZIP、现实例旧图片和从 ZIP 校验出的新图片会短暂同时存在。保守估算应至少
预留“ZIP 文件大小 + 当前 `/data/uploads` 大小 + 备份内图片大小”的可用空间。图片很多时，
先扩容卷再恢复。

## 从网站恢复

1. 打开“设置 → 数据与备份”，选择 ZIP。
2. 输入当前网站密码并确认恢复。
3. 等待成功提示，不要刷新或重复提交。
4. 检查余额、最近任务、奖励、历史图片和称呼。
5. 让 AI 重新调用 `get_overview`，再进行新的写操作。
6. 确认无误后再决定是否删除恢复前的卷快照。

上传文件先落在 `/data/tmp`，而不是进入进程内存；请求结束或下次启动时会清理临时上传。
恢复会逐项执行以下校验：

- ZIP 总大小、entry 数量、manifest 大小和解压后总量；
- manifest 格式、表名、字段、行数和关键单例记录；
- 归档路径与数据库对象 Key 白名单，拒绝绝对路径和目录穿越；
- 图片大小、SHA-256、WebP 真格式、像素和宽高。

图片会先写入服务端生成的新对象 Key。全部文件校验通过后，业务表与幂等记录才在一个数据库
事务中替换；事务提交后再删除旧图片。校验或事务在提交前失败时，原数据库和旧图片保持不变。
若进程在极短的切换窗口中意外退出，下次启动会清理没有数据库引用的图片目录。

## 限制与失败处理

- 上传 ZIP 最大 2 GiB；
- manifest 最大 32 MiB；
- 单个归档图片对象最大 64 MiB；
- 解压后总量最大 4 GiB；
- 同一时间只允许一个导出或恢复操作。

这些上限是安全边界，不代表接近上限的文件一定适合当前卷容量或实例规格。恢复返回
`invalid_backup` 时不要反复尝试同一来源不明的文件；保留原卷和错误信息。若恢复过程中容器
因磁盘不足退出，先扩容或回挂恢复前快照，再重新操作。

## 建议周期

- 每次升级前：创建 `/data` 卷快照并下载一次网站 ZIP；
- 每周：把网站 ZIP 保存到独立设备或云盘；
- 每月：在隔离的临时实例演练一次 ZIP 恢复；
- 重大任务、积分或奖励调整前：额外创建一个恢复点。

SQLite 使用 WAL。不要在应用运行时只复制 `phosphene.sqlite` 单文件；WAL 中可能还有未合并
事务。手工复制整套数据库文件前必须正常停止应用，通常优先使用平台卷快照或网站 ZIP。
